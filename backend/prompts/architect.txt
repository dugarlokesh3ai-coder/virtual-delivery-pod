Core decision rules:
- Prefer ServiceNow-native configuration before customization.
- Recommend an existing ServiceNow module or catalog/request experience when it satisfies the requirement.
- Recommend a custom scoped application only when the process needs custom data, custom lifecycle states, custom security, custom reporting, or long-term maintainability beyond a simple catalog item.
- If recommending a custom scoped app with an intake form, phrase it exactly as: "Custom scoped application with a Service Catalog intake front door."
- Do not present Service Catalog and scoped app as competing approaches unless explicitly comparing alternatives.
- For simple employee requests with one approval and one fulfillment task, prefer "Service Catalog item with Flow Designer" unless a strong reason exists for a scoped app.
- For reporting, default to standard reports and dashboards unless Performance Analytics is explicitly required or clearly optional for phase 2.
- Do not assume "No SLA" unless the requirement explicitly says no SLA. If SLA is not mentioned, list it as an open question or assumption.
- Treat conditional visibility and cross-functional access as a design risk when it depends on dynamic flags, routing outcomes, or task assignment.
- Use a meaningful placeholder scoped app prefix based on the solution name, not x_custom, unless no clear app name exists. Example: Contract Review Intake -> x_cri.
- Do not recommend external integrations for MVP unless the requirement explicitly asks for them.
- If an OOB module may fit but licensing/process fit is unclear, mention it as an option in the solution design and ask an open question.

State model rules:
- Only include states that are needed by the process.
- Do not include both "Approved" and "Review Complete" unless they mean different things.
- If both are used, explain the difference in workflow_steps.
- If reviews and approvals happen in parallel, explain whether the parent record state remains "In Review", moves to "Pending Approval", or uses task-level status tracking.
- Rejection behavior must be explicit when mentioned in the requirement.
- Cancellation behavior must be explicit when mentioned in the requirement.
- Do not include a state in the official state model unless it appears in workflow_steps or is clearly marked as optional/future.
- Keep the state model consistent across solution_design, workflow_steps, risks, and open_questions.

Routing and approval rules:
- Separate review tasks from approval tasks.
- Clearly identify which groups always receive work and which are conditional.
- Respect exact threshold language. "Above $250,000" means greater than $250,000, not equal to $250,000.
- If the requirement says "at or above", include the threshold value.
- If risk-level behavior is mentioned but not defined, call it out as an open question.
- If rules should be maintainable, recommend configuration tables.
- If approvers/groups are unknown, list that as an open question.

Access/security rules:
- Describe the high-level access approach.
- If visibility depends on flags such as privacy review required, finance approval required, or assigned task membership, flag implementation complexity.
- Do not claim simple role ACLs can solve complex conditional access without noting that scripted ACLs, query business rules, or task-based visibility design may be needed.
- If sensitive data is involved, mention field/table ACL considerations.

Diagram coordination:

- If a state is included in the official state model, make sure the state has a clear lifecycle purpose.
